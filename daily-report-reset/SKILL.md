# 日报画像清空

清空 `~/.daily-report/profile.json` 的部分或全部字段,让日报回到对应的 L1/L2 状态。

## 执行流

### Step 1:读现有画像

Read `~/.daily-report/profile.json`。

- 文件不存在或读取失败 → 告诉用户「你还没有设置过日报画像,无需清空。如果想配置画像,说'设置日报'即可。」→ **直接结束**,不进入后续步骤
- 读取成功 → 进入 Step 2

### Step 2:判断清除粒度 + 展示影响

**先从用户这一轮消息里判断粒度**:

| 用户说的话 | 粒度 |
|-----------|------|
| "全部清除"/"全清"/"重置"/"删除画像"/"从头再来" | `all` |
| "只清持仓"/"清掉持仓" | `positions` |
| "只清自选"/"清掉自选" | `watchlist` |
| "清掉偏好"/"重置偏好" | `preferences` |
| "清空日报"/不明确 | 让用户选 |

**用自然语言展示当前画像内容 + 选项**(不要展示 JSON):

示例:
> "你当前的日报画像:
> - 关注 A 股、美股
> - 自选 3 只(茅台 / 英伟达 / 芯片 ETF)
> - 持仓 3 只共 ¥98,000
> - 偏好简洁直接
>
> 你想清除哪部分?
> 1. **全部清除**——回到 L1 通用日报
> 2. **只清持仓**——保留自选,不再计算盈亏
> 3. **只清自选**——清掉自选和持仓,只看大盘
> 4. **只清偏好**——风格恢复默认"

如果用户这一轮已经明确说过粒度(如"只清持仓"),跳过选项列表,直接确认:
> "要把持仓全部清除,保留自选和偏好。确认吗?"

### Step 3:确认后执行

**必须等用户确认**再执行。破坏性操作不能跳过确认。

根据粒度执行:

#### 3.1 all(全部清除)

Write 入空状态(保留 schema 结构,**不删文件**):

```json
{
  "version": 1,
  "created_at": "保留原 created_at",
  "updated_at": "当前时间 ISO",
  "L2": { "markets": [], "watchlist": [] },
  "L3": { "positions": [] },
  "preferences": { "style": null, "focus": null, "length": null }
}
```

#### 3.2 positions(只清持仓)

Read 原 profile → `L3.positions = []` → 其他字段保留 → 刷新 `updated_at` → Write 回

#### 3.3 watchlist(只清自选)

Read 原 profile → `L2.watchlist = []` **且** `L3.positions = []`(联动)→ 其他字段保留 → 刷新 `updated_at` → Write 回

> ⚠️ 清自选必须联动清持仓,因为持仓依赖 watchlist 的 symbol。

#### 3.4 preferences(只清偏好)

Read 原 profile → `preferences.{style, focus, length}` 全设为 null → 其他字段保留 → 刷新 `updated_at` → Write 回

### Step 4:告知结果

根据清空后的新 level 告知用户:

| 清空后状态 | 提示 |
|-----------|------|
| L1(watchlist 和 positions 都空) | "已全部清空。下次生成日报将是通用版本(L1)。" |
| L2(watchlist 非空,positions 空) | "已清空持仓。日报会保留自选分析,不再计算盈亏(L2)。" |
| L2(只清偏好) | "已清空偏好,风格恢复默认。自选和持仓没动。" |

## 行为规则

### 必须确认
- 任何破坏性操作**必须**先展示影响并等用户确认
- 用户说"不"/"算了"/"取消" → 立即中止,告知"好的,画像没改动"

### 展示影响
- 让用户清楚知道清空后会失去什么
- 用自然语言,不要给 JSON

### 不误伤
- 用户说"清空"但没指明粒度 → 列选项让用户选,**不要默认全清**
- 清自选时自动联动清持仓,告诉用户这是联动

### 不额外操作
- 清完就结束,不要主动引导用户重新设置
- 如果用户想继续配置,他自己会说"设置日报"

## 异常处理

| 场景 | 处理 |
|------|------|
| profile.json 不存在 | 在 Step 1 结束,提示"无需清空" |
| profile.json JSON 非法 | 告知用户"画像文件格式异常,直接覆盖为空状态?"→ 用户确认 → Write 空状态 |
| 用户说了一个不存在的字段要清 | 告知"画像里没有这个字段,要清的应该是 ___ 对吗?" |
